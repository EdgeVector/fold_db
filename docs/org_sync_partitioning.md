# Organization Sync Partitioning

This document explains the architecture of the **Organization Data Sync Partitioning** inside FoldDB. It dictates how private personal data and shared corporate data elegantly co-exist in the same local database, while correctly synchronizing to totally distinct storage buckets in the cloud.

## Overview

FoldDB operates on a unique "Thin Shell" multi-tenant model. An individual user and the organizations they belong to all utilize the same underlying local `Sled` and `DynamoDB` structures. 

To maintain strict isolation, FoldDB relies on a totally lockless **Key Prefix Partitioning** system in the database storage layer. This architecture removes the complexities of mapping tables, guaranteeing that personal data never leaks to an organization bucket, and vice-versa.

## 1. Atoms are Content, Not Storage
Atoms are defined entirely by their schema and contents. The UUID of an Atom (`uuid-X`) is deterministically generated by hashing its `source_schema_name` and its JSON payload recursively.

At the conceptual layer, there is no difference between a personal atom and an organizational atom. Identical data yields an identical Atom.

## 2. Storage Prefixing
When data reaches the physical database boundary (via `DbOperations` in `atom_operations.rs`), FoldDB checks if the Schema belongs to an organization (`org_hash = Some(hash)`).

If the data is deemed organizational, the database injects the Organization Hash into the physical storage key right before it is written:

* **Personal Atom Key:** `atom:uuid-X`
* **Organizational Atom Key:** `abc123def456:atom:uuid-X`

Because the underlying Key-Value store handles keys literally, it stores them as two separate entries. This allows identical content to concurrently exist in personal and organizational scopes without conflict, eliminating multi-tenant data collisions at the foundation.

## 3. The Sync Partitioner
The Sync Engine handles synchronization from the local append-only log up to the cloud (`org_sync.rs`). Instead of checking access lists, it simply inspects the physical keys written in the log:

```rust
// A simplification of routing logic
if key.starts_with(&"{org_hash}:") {
    // 1. Encrypt with Organization's E2E Key
    // 2. Upload to S3: `/{org_hash}/log/{member_id}/...`
    return SyncDestination::Org;
}

// 1. Encrypt with User's Personal E2E Key
// 2. Upload to S3: `/{user_hash}/log/...`
return SyncDestination::Personal;
```

This ensures that the routing rules are self-describing. If a bug ever causes data to be stored without an organization prefix, it simply falls back to the high-security personal sync path—meaning data will never accidentally leak into a public organization bucket.

## Summary

This stateless, prefix-driven architecture keeps the `fold_db` synchronization process exceptionally thin and scalable. The local database inherently knows what belongs where strictly by how the keys are padded.
