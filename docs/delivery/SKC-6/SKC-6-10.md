# SKC-6-10 Remove legacy fallback logic from universal key resolution

## Description
Remove the `create_legacy_resolved_keys` fallback function and related fallback logic that was introduced during SKC-6-2 implementation. This will enforce strict schema-driven key extraction and ensure all field processing paths require proper schema configuration.

## Requirements
- Remove the `create_legacy_resolved_keys` function from `field_processing.rs`
- Remove the fallback logic in `resolve_universal_keys` that catches errors and falls back to legacy extraction
- Update `create_molecule_for_field` to propagate errors from `resolve_universal_keys` instead of falling back to legacy behavior
- Ensure all schemas must be properly configured with universal key settings or explicit legacy field names
- Update any tests that depend on the legacy fallback behavior

## Implementation Plan
1. Remove the `create_legacy_resolved_keys` function and its implementation
2. Update `resolve_universal_keys` to propagate errors instead of falling back to legacy extraction
3. Update `create_molecule_for_field` to handle errors from `resolve_universal_keys` appropriately
4. Review and update tests that may have been relying on the legacy fallback behavior
5. Ensure all schemas in tests have proper key configuration or use explicit legacy field names

## Test Plan
- Run `cargo test --workspace` to identify any tests that were relying on the legacy fallback
- Update failing tests to use proper schema configurations instead of relying on fallback behavior
- Verify that tests with missing schemas properly fail with descriptive error messages
- Run `cargo clippy --all-targets --all-features` to ensure no unused code remains

## Verification
