## What is needed

To unblock TH6a, the canonical spec needs to land in exemem-workspace and answer at least the
following questions. The fold_db work this PR was meant to do is mechanical once they are pinned.

### Open questions the spec must close

1. **`max_input_bytes` definition.** The task references `snapshot_truncated: bool — true when the input
   exceeded max_input_bytes and was truncated`, but never specifies the value, where it is configured
   (per-view? global?), or what truncation looks like for a `BTreeMap<String, Vec<Value>>` envelope
   (drop entries? truncate strings within values? a top-level byte cap on the serialized JSON?).
   Each answer leads to a materially different implementation.

2. **`ContentHash` type.** `schema_versions: HashMap<String, ContentHash>` — fold_db has
   `Schema::compute_identity_hash` (returns a `String`), and view derivation uses a 64-char hex digest
   (`wasm_hash`, `input_snapshot_hash` on `DerivedMetadata`). The spec needs to pin which one TH6
   serializes to the row, and whether the field is `String` or a newtype. Picking one now would lock
   the wire format for downstream consumers (TH6b export, fold_db_node CLI).

3. **Retention bucket semantics.** The task says retention is `{ errors, all_with_snapshot, metadata_only }`
   with caps 100/10/1000, and that "Snapshots evict before metadata; the firing row's metadata persists
   longer than its snapshot." But:
   - Are these per-view caps or per-trigger-id caps?
   - Is "metadata_only" the bucket of rows whose snapshot has been evicted, or a separate bucket of
     rows that never had a snapshot (e.g., successful firings under `ErrorsOnly`)?
   - Is eviction synchronous (on every write) or swept (background task)? The TriggerFiring schema is
     a `HashRange` keyed by `(trigger_id, fired_at_ms)` — there is no in-place "null out the snapshot
     field" operation today; clearing a single field on an existing row would require either a new
     mutation type or rewriting the row, both of which are scope decisions the spec needs to make.
   - Does evicting a snapshot delete it via a `MutationType::Update` that overwrites `input_snapshot`
     with Null, or is there a separate snapshot store keyed off the firing row id?

4. **Storage shape for the snapshot.** `input_snapshot: Option<JsonObject>` — the existing
   `FieldValueType` has `Object` via `Json`/`Any` variants, but the schema service rejects schemas
   whose fields are typed `Object` without explicit nested type info. The spec needs to confirm
   whether the snapshot field is:
   - a single `FieldValueType::Json` blob field on `TriggerFiring` (simplest), or
   - a sibling schema (e.g. `TriggerFiringInput`) keyed off `(trigger_id, fired_at_ms)` so retention
     can drop snapshot rows without rewriting the parent row.
   The "snapshots evict before metadata" requirement strongly implies the second shape, but that is
   a much larger PR (new schema, registration, dual-write, the view orchestrator's fire path needs
   to thread the input through to the writer, etc.).

5. **Where `firing_capture` / `firing_retention` live on the view.** `TransformView` is the
   user-facing definition (`crates/core/src/view/types.rs:98`). The schema service validator imports
   the trigger types from fold_db and must round-trip the same shape (`crates/core/src/triggers/types.rs:1-7`).
   Adding the new fields here changes the canonical view contract and needs schema-service buy-in
   (the validator's `Trigger` shape is shared; whether per-view config sits *on the trigger* or
   *alongside it* is a structural choice).

6. **Default for existing views.** Task says "existing views default to Off (preserves current
   behavior)" but new views default to "ErrorsOnly". The disambiguator must be deterministic — by
   schema-service registration version, by a serde-default flag, by a registry boot migration?
   `#[serde(default)]` on a new field defaults to its `Default` impl, not "Off vs ErrorsOnly based
   on age." Spec needs to pin the disambiguator.

7. **Schema content hash at firing time.** The task says
   `schema_versions: HashMap<String, ContentHash> — schema name to schema content hash captured at
   firing time`. `SchemaCore` has `compute_identity_hash` on the schema struct; whether this is the
   right hash for the wire (vs `compute_content_hash` if such a thing is added) and whether it's
   cheap enough to recompute per fire (vs cached on registration) is a perf question the spec
   should at least gesture at.

### What I did NOT do (deliberately)

- Did **not** invent answers to the above and ship a partial implementation. The downstream
  TH6b consumer (fold_db_node CLI export, separate task) reads whatever wire format we land here.
  Picking a format now without spec backing creates a migration we'd own.
- Did **not** write tests, since they would pin the same invented answers.
- Did **not** modify `TriggerFiring` schema or `FiringRecord`, since the field shape is the
  load-bearing decision.

## Diagnostic details

### Where the work would land (mapped during scope review)

- `crates/core/src/triggers/mod.rs` — TriggerFiring schema definition, ~277 lines. Adding
  `input_snapshot`, `schema_versions`, `snapshot_truncated` field constants and types here.
- `crates/core/src/fold_db_core/trigger_runner.rs` — 3151 lines. `FiringRecord` (line ~117),
  `into_mutation` (line ~185), `FireOutcome` (line ~82). All extend.
- `crates/core/src/fold_db_core/view_orchestrator.rs:240` — `fire_view` is where input is built
  and WASM is called via `ViewResolver::resolve_with_overrides_and_derived`. The resolver builds
  `all_query_results: HashMap<String, HashMap<String, HashMap<KeyValue, FieldValue>>>`
  (`crates/core/src/view/resolver.rs:151-159`) — this is the "WASM input envelope" the task
  refers to. It is **not** currently surfaced back to the orchestrator; resolver returns only
  `(output, Option<DerivedMetadata>)`. Surfacing it requires extending the resolver's return
  signature *or* adding a new "with snapshot" entry point. Scope decision the spec should make.
- `crates/core/src/view/types.rs:98` — `TransformView`. Where `firing_capture` /
  `firing_retention` would attach.
- `crates/core/src/triggers/types.rs:1-7` — `Trigger` enum. Header explicitly notes
  schema_service shares this shape. Any field we add round-trips into the validator.

### Existing facilities we'd build on

- `Schema::compute_identity_hash()` exists and produces a stable hash on the schema struct.
- `DerivedMetadata` (`crates/core/src/view/derived_metadata.rs`) already computes
  `input_snapshot_hash` (64-char hex SHA-256 of the input envelope) and `wasm_hash`. This is a
  hash, not the snapshot — but the hashing path is shared and the snapshot envelope itself is
  built right next to it.
- `MutationType::Create` is what `FiringRecord::into_mutation` uses today. Eviction of a single
  field on an existing row would need `MutationType::Update` against the TriggerFiring schema,
  which the runner does not currently emit.

## How to unblock

Pick one of these paths, in order of preference:

1. **Land the spec in exemem-workspace.** Author `docs/designs/transform_test_harness.md` with the
   "TriggerFiring snapshot spec" section answering the 7 open questions above. Re-fire this task
   and it lands cleanly.

2. **Reduce scope to TH6a-pre.** Land *only* the wire fields (the three new TriggerFiring fields,
   defaulted to Null/empty/false) and `firing_capture: Off`-only on every view. No retention, no
   snapshot capture path. This is mechanically safe and unblocks downstream TH6b's read shape
   without committing to capture/retention semantics. Re-fire with that scope and it ships.

3. **Land the canonical spec inline in this PR.** Author the spec doc as part of TH6a so it
   ships with the implementation. Heavier but keeps the work in one place.
